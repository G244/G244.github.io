redis 集群产生的背景

 主从-哨兵:能解决数据备份以及redis的高可用,
 但是不能解决一个问题就是数据的容量,一旦数据过大
 一台redis服务器是抗不住的
 所以就诞生了redis cluster集群来分配多个redis服务器

3 个主节点 3个从节点
集群的好处：将数据切分为多个节点的能力
           当一部分节点失去通讯时,仍然可以继续处理命令请求的能力,拥有自动故障转移的能力
数据分布分为: 顺序分区 | 哈希分区
顺序分区:是以全量数据进行顺序平均的分区规则分给各个子集 会导致数据倾斜
哈希分区:是以hash函数进行取余产生的数 假如有3台则余3 所以不好扩容     
        取余点分区: 
        一致性分区: 为系统中每一个节点随机分配0~232的token构成一个哈希环,
                   然后数据读写的时候根据key计算hash值,然后顺时针的找到第一个大于或者等于哈希值的token节点
        虚拟槽分区: 使用哈希函数把所有数据映射到一个固定范围内的整数集合中,整数定义为槽,redis cluster槽的范围是
                   0~16383,槽是集群内数据管理和迁移的基本单位.采用大范围的槽目的是方便数据拆分和集群扩展,一个节点对应着一定数量的槽

  总结:一个节点有多个槽,一个槽中有多个数据,如果要迁移数据就移动槽到另一个节点上    

 集群搭建过程：
  1.准备节点

         #节点端口
         port 6379
         #开启集群模式
         cluster-enabled yes
         #节点超时时间，单位毫秒
         cluster-node-timeout 15000
         #集群内部配置文件
         cluster-config-file "nodes-6379.conf"

  2.节点握手
        cluster meet ip port

1、每个Redis Cluster节点会占用两个TCP端口，一个监听客户端的请求，默认是6379，另外一个在前一个端口加上10000
redis-cli -h 192.160.1.10 cluster replicate 6099eed758b664bbe40c972c09769fa2c97cd91c

3.分配槽   
 # redis-cli -h 192.160.1.100 cluster addslots {0..5461}

对节点的通信在redis端口号上加10000


cluster nodes 
redis-cli --cluster create 192.160.1.200:6379 192.160.1.201:6379 192.160.1.202:6379 192.160.1.203:6379 192.160.1.204:6379 192.160.1.205:6379 --cluster-replicas 1
